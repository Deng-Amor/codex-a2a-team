# 项目底层规则与工程红线

本文件适用于本仓库内所有 Agent、Worker、分支与服务。与临时任务要求冲突时，以本文件为准；需要例外必须由用户明确确认并记录。

## 1. 技术栈边界

| 场景 | 强制技术栈 |
| --- | --- |
| Web 前端 | Vue 3 + Vite |
| 微信小程序、H5、App | uni-app |
| 后端 API | FastAPI + PostgreSQL |
| 缓存、分布式锁、队列 | Redis；仅在明确存在性能、并发、会话、队列或限流需求时接入 |
| HTTP API | RESTful；使用版本前缀，例如 `/api/v1` |

不得在未获明确批准时替换上述技术栈、引入第二套前端框架，或以临时脚本绕开正式后端 API。

## 2. API Contract 红线

1. Team Lead 必须先产出并通过审计的 API Contract，前后端才能开始实现。
2. Contract 至少定义：路径、HTTP 方法、鉴权、请求字段、响应字段、分页、枚举、成功码、错误码与示例。
3. 前端 Mock 数据、FastAPI Schema 和接口测试必须遵守同一份 Contract。
4. 前端可先基于 Mock 并行开发；后端完成后必须执行 Mock 到真实 API 的集成验证。
5. 不得让前端猜字段、让后端静默改字段，或用未记录的接口破坏兼容性。

## 3. MCP 使用规则

任何 Agent 判断需要新增、安装、调用或连接 MCP 服务时，必须先说明：服务名称、用途、读写权限、涉及的数据和替代方案，并等待用户确认。未经确认不得调用具备外部写入、副作用或凭据访问能力的 MCP。

## 4. Git、分支与 PR 红线

1. 所有代码变更必须在独立分支完成并推送远端。
2. 任何合并必须通过 Pull Request；禁止直接向受保护分支推送或合并。
3. PR 必须关联任务、说明变更范围、验证结果、数据库迁移和回滚影响。
4. 简单 Bug 修复和低风险 PR 在自动检查、审计和测试均通过后可自动合并，无需人工确认；其余 PR 必须提示用户 Review 并由用户确认合并。
5. 不得覆盖他人未合并工作、强推共享分支、跳过审计或伪造测试结果。

## 5. 数据库与 SQLAlchemy 红线

1. PostgreSQL 是唯一正式关系型数据源；不得以本地 JSON 替代正式业务数据。
2. 任意表、字段、索引、约束、枚举或数据语义变更，必须同步更新 SQLAlchemy 模型和版本化迁移；推荐使用 Alembic 管理迁移。
3. 禁止只在某个环境手工执行 SQL 而不提交迁移文件。
4. 迁移必须可从空库执行，且应说明升级、回滚和数据兼容策略。
5. 多分支开发时，合并前必须检查迁移编号冲突、依赖顺序和应用兼容性。
6. 生产数据修复 SQL 必须独立评审、可审计、可回滚；禁止通过 ORM 自动建表替代正式迁移。

## 6. PostgreSQL 备份

1. 为满足 RPO ≤ 2 小时，至少每 2 小时执行一次 `pg_dump` 逻辑备份，输出目录为 `data/backups/`；可额外在每日 02:00 保留一份日归档。
2. 备份文件命名：`agent_to_agent_YYYYMMDD_HHMMSS.dump`，使用自定义格式以支持 `pg_restore`。
3. 备份任务使用环境变量提供连接信息；禁止把数据库密码写入脚本、文档或 Git。
4. 至少保留 30 天备份；清理策略必须只作用于 `data/backups/`。
5. 每月至少验证一次恢复演练：新建临时数据库、执行 `pg_restore`、运行健康检查并记录结果。
6. RPO（可容忍数据丢失）≤ 2 小时；备份失败必须告警，且在下一次备份窗口前修复。
7. 简单故障 RTO（恢复时间目标）≤ 2 小时；复杂故障应对受影响功能启用“修复中”降级页，禁止展示错误堆栈，并在 24 小时内完成修复或升级处置。

## 7. A2A 编排与质量闸门

1. Team Lead 维护共享任务列表、上下文摘要、风险、API Contract 和动态 DAG。
2. 方案审计通过前，开发 Agent 不得开始；涉及前后端时，审计通过后才允许前后端并行。
3. 每个节点必须记录：输入、具体任务、执行日志、产物、交接摘要、关联消息和验收结果。
4. 子任务要能独立验收；边界不清时退回 Team Lead，不以增加 Agent 数量掩盖不确定性。
5. 默认并发规模控制在 3–5 个 Agent；超出时需说明并发收益与成本。
6. 每个任务设置最大迭代次数和重复消息检测；超过阈值升级 Team Lead 或人工确认。
7. 架构/API Contract、数据库迁移、自动合并、生产部署和高风险操作必须保留人工确认点。

## 8. 安全与可观测性红线

1. 密钥、Token、密码、连接串不得提交 Git 或写入日志；Dashboard 如必须展示配置，只可展示脱敏值，例如 `postgres://user:***@host:5432/db`。
2. 所有外部写操作必须记录操作者、Agent ID、Codex Session ID、Workflow/Task ID、分支、Commit、PR、时间、目标和结果；由此可从任意产物反查责任人和审批记录。
3. API 必须实施鉴权、输入校验、统一错误响应与最小权限原则。
4. 失败、重试、回滚和人工介入必须在工作流日志中可追溯。
5. 不得把“已创建任务”“计划产物”描述为“已交付”；只有节点通过验证并写入实际交付证据后，才能标记 `passed`。
6. 回滚不等同于 Git 回退：发布必须同时具备代码版本、部署制品、数据库迁移兼容性和数据回滚/补偿方案；不得只回退代码而留下不兼容数据库结构。
